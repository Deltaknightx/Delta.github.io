1989–1991

Origins of HTTP

The Hypertext Transfer Protocol began at CERN when Sir Tim Berners-Lee proposed the World Wide Web. His early work included a simple client, a server, and a lightweight protocol for transferring hypertext documents. This early design would become HTTP/0.9.

  • 1989: Proposal for a distributed information system
  • 1990: First browser and server prototypes built
  • 1991: Public release of the early Web project
The early versions of HTTP were intentionally simple to encourage adoption and experimentation.
HTTP/0.9

HTTP/0.9 — The One-Line Protocol

HTTP/0.9 was extremely simple and designed only for transferring raw HTML. It supported just one method, GET, and responses contained no headers — only HTML content.

  • Single method: GET
  • No status codes
  • No headers
  • Text-based responses only
HTTP/0.9 pages could not include images, styling, or scripts — only plain HTML.
1996–1997

HTTP/1.0 and HTTP/1.1 — Growing the Web

As the Web exploded in popularity, HTTP needed richer features. HTTP/1.0 introduced status codes, headers, and support for more media types. Shortly after, HTTP/1.1 standardized persistent connections, caching rules, and chunked transfer encoding.

  • HTTP/1.0 — Introduced headers, MIME types, basic caching, and status codes.
  • HTTP/1.1 — Added persistent connections, chunked transfers, and better caching.
Most of the modern Web still uses HTTP/1.1, especially for APIs and legacy systems.
Motivation

Why the Protocol Needed Improvement

The early Internet was drastically slower than today. Using many TCP connections for every image, script, or style file caused delays. As pages grew more complex, browsers and servers needed ways to reduce latency and reuse existing connections.

  • Growing demand for multimedia content
  • Need for reusable connections
  • Bandwidth limitations
  • More metadata required for dynamic content
These limitations pushed developers toward new designs, eventually influencing HTTP/2 and HTTP/3.
